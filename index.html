If you have ever clicked a "Pay Now" button twice and panicked, you already understand why idempotency matters. In distributed systems, network hiccups, client retries, and impatient users routinely send the same request more than once. Without idempotency, each duplicate request creates real damage — double payments, duplicate database rows, or conflicting state.

In this guide, you will learn what idempotency is, why your Spring Boot application needs it, and how to implement it step by step. You will also discover common pitfalls that catch even experienced developers off guard.

What Is Idempotency and Why Should You Care?

An operation is idempotent when performing it once produces the same outcome as performing it multiple times. HTTP methods like GET, PUT, and DELETE are idempotent by design. POST, however, is not — and that is exactly where trouble starts.

Consider this real-world scenario: a customer submits a payment. The server processes the charge successfully but the response times out before reaching the client. The client retries the exact same request. Without idempotency protection, your application happily processes a second charge.

⚠

The result? An angry customer, a support ticket, and a refund to process.

Idempotency eliminates this entire class of bugs by ensuring that retried requests return the original response without re-executing the business logic.

Without Idempotency: A Costly Mistake

Take a look at a typical payment endpoint that lacks idempotency protection:

Java 
@PostMapping("/payments")
public ResponseEntity<PaymentResponse> createPayment(@RequestBody PaymentRequest request) {
    Payment payment = paymentService.processPayment(request);
    return ResponseEntity.status(HttpStatus.CREATED).body(toResponse(payment));
}

Every time a client calls this endpoint, the service processes a brand-new payment. If the network drops the response and the client retries, the customer gets charged twice. There is no mechanism to detect that this request has already been handled.

❌

What goes wrong:

  • The customer gets billed twice (or more).
  • Your database fills up with duplicate records.
  • Downstream systems (email notifications, inventory updates) fire multiple times.
  • Reconciliation becomes a nightmare.

With Idempotency: Safe, Predictable, Resilient

The most widely adopted pattern uses an idempotency key — a unique identifier the client sends with each request. Your server stores the result of the first execution and returns it verbatim for any subsequent requests carrying the same key.

Step 1: Define the Idempotency Key Store

Start with a simple entity to persist processed requests:

Java — IdempotencyRecord.java 
@Entity
@Table(name = "idempotency_keys")
public class IdempotencyRecord {

    @Id
    private String idempotencyKey;

    private int responseStatus;

    @Column(columnDefinition = "TEXT")
    private String responseBody;

    private LocalDateTime createdAt;
}

Back this with a Spring Data repository:

Java — IdempotencyRepository.java 
public interface IdempotencyRepository extends JpaRepository<IdempotencyRecord, String> {
}

Step 2: Build an Idempotency Interceptor

Rather than scattering idempotency logic across every controller, centralise it in a servlet filter:

Java — IdempotencyFilter.java 
@Component
public class IdempotencyFilter extends OncePerRequestFilter {

    private final IdempotencyRepository repository;
    private final ObjectMapper objectMapper;

    public IdempotencyFilter(IdempotencyRepository repository, ObjectMapper objectMapper) {
        this.repository = repository;
        this.objectMapper = objectMapper;
    }

    @Override
    protected void doFilterInternal(HttpServletRequest request,
                                    HttpServletResponse response,
                                    FilterChain chain) throws ServletException, IOException {

        String key = request.getHeader("Idempotency-Key");

        if (key == null || !"POST".equalsIgnoreCase(request.getMethod())) {
            chain.doFilter(request, response);
            return;
        }

        Optional<IdempotencyRecord> existing = repository.findById(key);

        if (existing.isPresent()) {
            IdempotencyRecord record = existing.get();
            response.setStatus(record.getResponseStatus());
            response.setContentType("application/json");
            response.getWriter().write(record.getResponseBody());
            return;
        }

        ContentCachingResponseWrapper wrappedResponse =
                new ContentCachingResponseWrapper(response);
        chain.doFilter(request, wrappedResponse);

        IdempotencyRecord record = new IdempotencyRecord();
        record.setIdempotencyKey(key);
        record.setResponseStatus(wrappedResponse.getStatus());
        record.setResponseBody(new String(wrappedResponse.getContentAsByteArray()));
        record.setCreatedAt(LocalDateTime.now());
        repository.save(record);

        wrappedResponse.copyBodyToResponse();
    }
}

Step 3: Update the Payment Endpoint

Your controller stays clean — the filter handles everything transparently:

Java 
@PostMapping("/payments")
public ResponseEntity<PaymentResponse> createPayment(@RequestBody PaymentRequest request) {
    Payment payment = paymentService.processPayment(request);
    return ResponseEntity.status(HttpStatus.CREATED).body(toResponse(payment));
}
✔

Now, when a client sends the same Idempotency-Key header twice, the second request returns the cached response instantly. No duplicate charge. No duplicate database row. No drama.

Common Pitfalls to Watch Out For

Even with a solid implementation, several traps await the unwary.

1

Race Conditions on Concurrent Retries

If two identical requests arrive simultaneously, both might pass the "does this key exist?" check before either writes the record. Use a database unique constraint on the idempotency key column and handle the constraint violation gracefully. Alternatively, use distributed locking with Redis (SETNX) for high-throughput systems.

2

Never Expiring Idempotency Keys

Storing keys indefinitely bloats your database. Set a sensible time-to-live — 24 to 48 hours covers the vast majority of retry windows. Schedule a cleanup job to purge expired records:

Java — Scheduled Cleanup 
@Scheduled(cron = "0 0 3 * * *")
public void purgeExpiredKeys() {
    repository.deleteByCreatedAtBefore(LocalDateTime.now().minusHours(48));
}
3

Caching Error Responses

Should you cache a 500 Internal Server Error? Generally, no. If the server fails, the client should be free to retry and get a successful result. Only cache responses with 2xx or 4xx status codes — transient failures deserve a fresh attempt.

4

Ignoring Request Body Mismatches

A malicious or buggy client might reuse an idempotency key with a different request body. Validate that the incoming payload matches the original. Store a hash of the request body alongside the key and return a 422 Unprocessable Entity if they diverge.

5

Forgetting Distributed Environments

An in-memory cache breaks the moment you scale to multiple instances. Always use a shared store — your relational database, Redis, or a similar centralised solution — so every instance sees the same idempotency state.

When Should You Apply Idempotency?

Not every endpoint needs it. Focus on operations that create resources or trigger side effects:

  • Payment processing and financial transactions
  • Order placement and fulfilment triggers
  • Webhook receivers from third-party services
  • Any POST endpoint where a retry causes observable duplication

GET, PUT, and DELETE endpoints are naturally idempotent when implemented correctly, so they rarely need additional protection.

Key Takeaways

  • Idempotency protects your users and your data from the inevitable chaos of distributed systems.
  • Use a client-supplied Idempotency-Key header to uniquely identify each operation.
  • Centralise the logic in a filter or interceptor to keep your controllers clean.
  • Guard against race conditions, unbounded storage, cached errors, and payload mismatches.
  • Store idempotency records in a shared, persistent store to support horizontal scaling.

Ready to Level Up Your APIs?

Idempotency is one of those patterns that separates fragile prototypes from production-grade systems. Start by adding it to your most critical endpoints and expand from there.
